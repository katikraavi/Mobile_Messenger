## Summary

Establish the complete Flutter + Serverpod monorepo project structure with Docker Compose infrastructure. This foundational feature creates the directory layout, project initialization, and containerized development environment (backend + PostgreSQL) that all subsequent features will build upon. Developers can clone the repo and run `docker-compose up` to have a fully operational backend and PostgreSQL database running locally within 2 minutes. The Flutter app can launch on an emulator and connect to the backend with automatic retry logic for startup race conditions.

## Technical Context

**Language/Version**: Dart (frontend: Flutter SDK 3.10+, backend: Serverpod 2.1+)  
**Primary Dependencies**: Flutter SDK, Serverpod, Dart SDK, PostgreSQL 13+, Docker Compose  
**Storage**: PostgreSQL database with Docker volume persistence  
**Testing**: Flutter widget/integration tests, Dart unit tests (framework setup only)  
**Target Platform**: Android, iOS via Flutter (mobile-first), Serverpod backend (Docker container)  
**Project Type**: Mobile app + backend monorepo with shared Dart codebase  
**Performance Goals**: Services start in <2 minutes, health endpoint responds <30 seconds, new developer setup <15 minutes  
**Constraints**: Single `docker-compose up` command starts all services; Serverpod auto-handles migrations; no manual configuration required  
**Scale/Scope**: 4 feature modules + infrastructure (auth, profile, chat, invites deferred to subsequent features)

**Structure Decision**: 
- **Frontend**: `/frontend` contains complete Flutter mobile app with feature-based modular organization (auth, profile, chat, invites). Each feature has screens, widgets, and services organized together. Core utilities (models, services, widgets) in `lib/core/` for DRY principles.
- **Backend**: `/backend` contains Serverpod server with endpoints, services, and models organized by domain. Health check endpoint in dedicated `health.dart` for monitoring/liveness checks.
- **Infrastructure**: `docker-compose.yml` at repository root defines PostgreSQL + Serverpod services with automatic networking, port mapping (5432 for DB, 8081 for backend), and volume persistence.
- **Rationale**: Separate `/frontend` and `/backend` directories enable independent development workflows, clear code ownership, and potential future monorepo scaling while sharing Dart as common language.

### Technology Choices (Confirmed)

- **Framework**: Flutter + Serverpod (chosen for full-stack Dart development, code sharing, and hot reload development experience)
- **Database**: PostgreSQL 13+ (mature, open-source, Docker-friendly, excellent Dart ORM support via Serverpod)
- **Containerization**: Docker Compose (local development parity with deployment, single-command setup)
- **Architecture**: Monorepo with separate frontend/backend (clear boundaries, independent CI/CD, Dart ecosystem advantages)

### Key Dependencies

| Dependency | Purpose | Version | Rationale |
|------------|---------|---------|-----------|
| Flutter SDK | Mobile frontend framework | 3.10+ | Latest stable, Dart 3.0+ support |
| Serverpod | Backend framework | 2.1+ | Built-in PostgreSQL ORM, WebSocket support, hot reload |
| PostgreSQL | Relational database | 13+ | Stable, Docker support, Serverpod native integration |
| Docker Compose | Local orchestration | v2.0+ | Single-command startup, volume persistence |
| Dart | Language | 3.0+ | Unified codebase for frontend/backend |

### Docker Compose Strategy

- **Service 1: PostgreSQL** - Database container with named volume for persistence across restarts
- **Service 2: Serverpod** - Backend container with environment variables pointing to PostgreSQL
- **Networking**: Docker's default bridge network enables automatic service discovery (backend → postgres via hostname)
- **Startup Sequence**: PostgreSQL initializes first; Serverpod waits and attempts automatic migrations
- **Health Checks**: Serverpod `/health` endpoint for container orchestration; PostgreSQL accessible on 5432

### Flutter Development Environment

- **Platform**: Android (primary) and iOS via Flutter's platform abstraction
- **Development**: Android Emulator or physical device required (not included in `docker-compose up`)
- **Connection**: App connects to backend via HTTP/WebSocket to localhost:8081 (from emulator's perspective: host.docker.internal:8081 on Android Studio emulator)
- **Retry Strategy**: Exponential backoff (100ms, 500ms, 2s, 5s, 10s) handles Serverpod startup delays
